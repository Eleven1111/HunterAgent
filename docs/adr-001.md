# ADR-001: Thin Core, Shared Services, Buffered Experimental Sourcing

## Status

Accepted

## Context

The legacy proposal set contained three conflicting impulses:

- A compliance-aware thin-core product.
- A business-deep but over-scoped full workflow system.
- A broad productized plan that still needed a clearer differentiation strategy.

The key decision was whether candidate acquisition automation should become part of the main product truth path.

## Decision

We will build HuntFlow vNext as a dual-track system:

1. A mainline workbench and chat product on shared domain services.
2. An experimental sourcing lane that stores raw items in a review queue before any promotion into the candidate system.

Formal state changes remain explicit and auditable.

## Consequences

### Positive

- The MVP can ship without depending on collection automation.
- Chat and workbench remain consistent because they share service implementations.
- Experimental sourcing can progress independently behind a feature flag.
- Compliance and operational boundaries stay visible in the architecture.

### Negative

- There is upfront complexity in maintaining review states and promotion flows.
- Experimental collection does not deliver immediate "full automation" theatrics.
- A file-backed store can preserve pilot data, but it is still not a substitute for transactional database infrastructure.

## Rejected alternatives

- Direct-write sourcing into the business tables: rejected because it makes the highest-risk subsystem a dependency of the mainline product.
- Full ATS/CRM/ERP first: rejected because it delays differentiation and pilotability.
- Chat-only product: rejected because consultants need dense operational surfaces, not just conversational access.

## Follow-on implementation note

The current implementation now includes:

- `memory`, file-backed, and Postgres-backed store modes,
- a `StoreGateway` boundary so route/runtime code no longer depends on raw collection access,
- a store factory plus per-collection Postgres persistence and Redis runtime state for release-oriented deployment,
- a server-side auth session model that supports `HttpOnly` web sessions and bearer compatibility for API clients,
- a buffered experimental sourcing lane whose public contract is controlled manual import and review,
- pilot Phase 6 modules for phone screen, assessment report, interview coordination, and invoice-lite.

The next architecture step remains deeper operational hardening and release validation, not re-expanding prototype capture surfaces.

## Documentation and gate implications

This ADR requires template-level artifacts to stay synchronized with implementation:

- `docs/events.md` is the event contract across mainline, approval/audit, experimental sourcing, and Phase 6.
- `docs/er-core.md` is the core relationship map that prevents truth-path drift.
- `docs/user-manual.md` is the operator contract for the dual-track execution loop.
- `docs/ops-sop.md` is the reliability contract for single-machine and production operations.

Gate interpretation follows this ADR:

- Phase 0 gate expects explicit architecture/doc baselines (including event and ER artifacts).
- Phase 6 gate expects operational hardening evidence (audit replay plus backup/restore and rollback SOP).
